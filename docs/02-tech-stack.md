# 02 — Tech Stack

## Frontend

### React 18 + TypeScript
The UI framework. TypeScript is used in strict mode everywhere — no `any` types allowed.
React 18 gives us concurrent features and automatic batching.

### Vite
The build tool and dev server. Much faster than Create React App. Handles hot module
replacement, TypeScript compilation, and production bundling.

### React Flow (`@xyflow/react`)
The canvas library. Handles all the hard parts of a node-based editor:
- Rendering nodes and edges on a canvas
- Drag and drop for nodes
- Drawing connections between ports
- Pan and zoom
- Minimap
We build custom node components on top of it.

### Zustand + Immer
Global state management for the frontend. Zustand is a minimal store library.
Immer lets us write state updates in a "mutable" style while keeping state truly
immutable under the hood. Two stores exist:
- `workflow.store.ts` — the current workflow being edited (nodes, connections, selected node)
- `auth.store.ts` — the current logged-in user

### TanStack Query
Handles all server state — fetching from Supabase, caching, background refetching,
and mutations (create/update/delete). Every Supabase query is wrapped in a custom hook
that uses TanStack Query under the hood.

### Tailwind CSS
Utility-first CSS framework. All styling is done via Tailwind classes. A set of CSS
variables is defined in `index.css` to support theming (light/dark mode).

### Radix UI
Accessible, unstyled component primitives. We use these for complex interactive
components like Dialog, Dropdown, Tooltip, Select, Popover. They handle keyboard
navigation and accessibility so we don't have to.

### React Hook Form + Zod
Form management and validation. React Hook Form handles form state with minimal
re-renders. Zod schemas from `@workflow/validators` are used as the validation
resolver — the same schemas used on the backend.

### Monaco Editor
The embedded code editor used inside the Code node config panel. This is the same
editor that powers VS Code. Gives users syntax highlighting and autocomplete when
writing JavaScript for Code nodes.

### React Router v6
Client-side routing. Handles navigation between Dashboard, Editor, Credentials,
Executions, and auth pages.

### `@supabase/supabase-js`
The official Supabase client. Used for auth, database queries, realtime subscriptions,
and storage access from the frontend.

### Supporting libraries
- `lucide-react` — icon library
- `clsx` + `tailwind-merge` — conditional class name composition via the `cn()` utility
- `date-fns` — date formatting in execution logs
- `uuid` — generating node IDs on the client before saving
- `lodash-es` — deep cloning workflow JSON before save operations

## Backend (Supabase)

### Supabase Auth
Handles user registration, login, JWT issuance, and token refresh. Supports email/password
and OAuth (Google, GitHub). JWTs are automatically included in all Supabase client requests
and used by RLS policies to filter data per user.

### Supabase Postgres (PostgreSQL 15)
The primary database. All application data lives here. Extensions used:
- `uuid-ossp` — UUID primary key generation
- `pgcrypto` — AES-256 encryption for credential storage
- `pg_cron` — scheduling cron jobs inside Postgres
- `pg_net` — making outbound HTTP calls from Postgres (used by pg_cron to trigger edge functions)

### Supabase Edge Functions (Deno)
Serverless functions that run on Deno. These are the backend logic layer:
- Executing workflows and individual nodes
- Receiving webhooks
- Encrypting and decrypting credentials
- Handling OAuth callbacks
Imports use Deno-compatible URLs (deno.land/std and esm.sh).

### Supabase Realtime
WebSocket-based pub/sub built on Postgres logical replication. The frontend subscribes
to changes on the `execution_logs` table to stream live node-by-node results during
a workflow run.

### Supabase Storage
Object storage for:
- File outputs generated by workflow nodes
- Exported workflow JSON files

### Row Level Security (RLS)
Every table has RLS enabled. Policies use `auth.uid()` to ensure users can only
read and write their own data. This means even if someone gets the anon key, they
cannot access other users' data.

## Monorepo tooling

### pnpm + pnpm workspaces
pnpm is the package manager. Workspaces allow multiple packages in one repo to
reference each other by name (e.g. `@workflow/types`) without publishing to npm.
This is how the frontend and edge functions share TypeScript types.

### TypeScript (strict mode)
Shared base config in `tsconfig.base.json` at the root. Each package and app extends
this. Strict mode is on everywhere — `noImplicitAny`, `strictNullChecks`, etc.
